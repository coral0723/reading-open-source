# Button 컴포넌트 분석

## 1. 이 컴포넌트의 역할
이 컴포넌트는 HTML `<button>` 요소를 기반으로 한 컴포넌트다.

## 2. 구현에서 눈에 띄는 점
하나의 Button 컴포넌트가 아닌,  
"행동"과 "표현"이라는 두 개의 서로 다른 책임을 가진 두 컴포넌트의 조합이 눈에 띄었다.  

### ① `@base-ui`의 Button.tsx
> 역할: 버튼이 **어떻게 행동해야 하는지**를 정의한다.  

**이 파일이 책임지는 것**
- 실제 `<button>` 요소 렌더링
- disabled / focusableWhenDisabled 처리
- 키보드 접근성
- ref 병합 관리
- ARIA / interaction 로직

📌 UI, 스타일, 디자인은 전혀 없음  
`import { Button as ButtonPrimitive } from "@base-ui/react/button"`
> 👉 그래서 이 컴포넌트는 **디자인 버튼이 아니라 "Primitive"** 에 가깝고,  
shadcn의 `Button.tsx`에서는 의도를 드러내기 위해 `ButtonPrimitive`라는 이름으로 사용된다.  

### ② `shadcn`의 Button.tsx  
> 역할: 버튼이 **어떻게 보일지, 어떻게 쓰일지**를 정의한다.  

**이 파일이 책임지는 것**  
- variant / size 정의 (cva)
- 디자인 시스템 규칙
- className 조합
- Base UI Button을 감싸는 인터페이스 제공

📌 동작 로직 없음
> 👉 **보여지는 버튼**에만 집중한 컴포넌트  

## 3. 왜 이렇게 설계했을까?
만약 이 둘이 하나의 컴포넌트였다면,  
로직들과 이벤트, 스타일 분기, 디자인 시스템 등이 전부 하나의 파일에 섞여 유지보수가 어려웠을 것이다.  

그래서 접근성과 동작은 검증된 `base/ui`에 위임하고,
`shadcn`은 UI와 디자인만 통제하기 위해 이런 구조의 설계를 택했을 것이다.  

## 4. 내가 만약 직접 만들었다면?
처음부터 이 구조를 떠올리기는 쉽지 않았을 것이다.  
이미 내가 만든 라이브러리에서는 하나의 컴포넌트에 로직과 디자인이 섞여있는 구조로 컴포넌트를 제공하고 있으니 말이다.

## 5. 이 코드에서 배운 점
- 동작 로직과 디자인 로직을 분리하는 방식은 확장성과 유지보수성을 크게 높인다
